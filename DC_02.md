# Data Communications — 2. 네트워크 모델
## 2.1 프로토콜 계층화
- 서비스 구현을 각 기능별로 분리
- 중간 시스템을 단순화
| | 내용 |
|---|---|
| 장점 | 기능별 분리, 중간 시스템 단순화 |
| 단점 | 계층 간 상호 연관이 있을 경우 정보 교환 어려움 |

### 기본 용어
- **노드 (Node)**: 네트워크에 연결된 장치 (ex: PC, 라우터, 스위치)
- **패킷 (Packet)**: 데이터를 잘게 쪼갠 전송 단위. 헤더(목적지 등 정보) + 데이터로 구성

## 2.2 OSI 모델
모든 유형의 컴퓨터 시스템 간 통신을 허용하는 네트워크 설계를 위한 **7계층 구조**
### OSI 7계층
| 계층 | 이름 | 핵심 기능 | 전달 단위 |
|---|---|---|---|
| 7 | 응용층 (Application) | 사용자 인터페이스, 네트워크 접근 | - |
| 6 | 표현층 (Presentation) | 암호화/복호화, 데이터 압축, 구문·의미 변환 | - |
| 5 | 세션층 (Session) | 세션 관리, 동기화, 대화 제어 | - |
| 4 | 전송층 (Transport) | 프로세스 대 프로세스 전달, 포트 주소, 분할/재조립 | 세그먼트 |
| 3 | 네트워크층 (Network) | 발신지-목적지 전달, 논리 주소, 라우팅 | 패킷 |
| 2 | 데이터 링크층 (Data Link) | 노드 대 노드 전달, 물리 주소, 흐름·오류 제어 | 프레임 |
| 1 | 물리층 (Physical) | 비트 스트림 전송, 전기·기계적 특성 | 비트 |
### 계층별 상세
**1계층 물리층 (Physical Layer)**
- 비트 스트림을 전자기 신호로 변환하여 매체를 통해 전송
- 기계적·전기적 특성 담당 (케이블, 커넥터)
> **비트 스트림**: 비트(0과 1)가 연속으로 나열된 데이터 흐름

**2계층 데이터 링크층 (Data Link Layer)**
- 노드 대 노드(hop-to-hop) 전달 책임
- 프레임 구성, 물리 주소 지정, 흐름·오류·접근 제어
  
**3계층 네트워크층 (Network Layer)**
- 발신지-목적지 패킷 전달 책임 (데이터 링크층은 노드 간)
- 논리 주소 지정 (IP), 라우팅
  
**4계층 전송층 (Transport Layer)**
- 전체 메시지의 프로세스 대 프로세스 전달
- 포트 주소 지정, 분할/재조립, 연결 제어, 흐름·오류 제어
  
**5계층 세션층 (Session Layer)**
- 세션 관리, 동기화, 대화 제어, 원활한 종료
  
**6계층 표현층 (Presentation Layer)**
- 두 시스템 간 데이터 구문(syntax)·의미(semantic) 처리
- 암호화/복호화, 데이터 압축
  
**7계층 응용층 (Application Layer)**
- 사용자가 네트워크에 접근할 수 있도록 인터페이스 제공
- 원격 로그인, 파일 전송, 메일, WWW
  
### OSI 모델의 실패 이유
- TCP/IP가 이미 완전히 자리잡은 후 OSI 완성
- 일부 계층이 완전히 정의되지 않음
- 전환 필요성을 충분히 보여주지 못함
> 현실에서는 OSI 모델은 개념/교육용으로만 쓰이고, 실제 네트워크는 TCP/IP 프로토콜을 사용함

## 2.3 TCP/IP 프로토콜
OSI 7계층에서 **세션층과 표현층 없음** → 5계층 구조
| TCP/IP 계층 | OSI 대응 | 주소 종류 | 설명 |
|---|---|---|---|
| 응용층 (Application) | 응용 + 표현 + 세션 | 이름 (Name) | 도메인 등 사람이 읽는 주소 |
| 전송층 (Transport) | 전송 | 포트 주소 (Port) | 특정 프로세스 식별 |
| 인터넷층 (Internet) | 네트워크 | 논리 주소 (IP) | 발신지-목적지 식별 |
| 데이터링크층 (Data Link) | 데이터 링크 | 물리 주소 (MAC) | 노드 간 홉 식별 |
| 물리층 (Physical) | 물리 | - | 주소 개념 없음 (신호 전송만 담당) |

### 주요 개념
- **데이터그램 (Datagram)**
  - 인터넷층(IP)에서 사용하는 독립적 전송 단위
  - 헤더(출발지/목적지 IP 등 제어 정보) + 페이로드(실제 데이터)로 구성
  - 패킷마다 경로가 달라질 수 있고 순서 보장 없음 → 수신측에서 재조립
  - 신뢰성/순서 보장은 상위 전송층(TCP)이 담당

- **MAC 주소 (Media Access Control Address)**
  - 제조사가 하드웨어에 고정으로 부여한 48비트 물리 주소 (ex: `00:1A:2B:3C:4D:5E`)
  - 앞 3바이트는 제조사 ID, 뒤 3바이트는 장치 고유번호
  - 이론상 전 세계 유일

- **포트 번호 (Port Number)**
  - 같은 IP 내에서 어떤 프로세스(앱)에게 데이터를 전달할지 식별하는 번호 (0~65535)
  - IP가 건물 주소라면 포트는 호실 번호
  - ex: HTTP→80, HTTPS→443, SSH→22, DNS→53

### 데이터 구조 (캡슐화)
실제 데이터는 계층별로 **양파처럼 겹겹이 감싸진 형태**로 전송됨

| 계층 | 단위 | 구조 |
|---|---|---|
| 전송층 | 세그먼트 | `출발지 포트 \| 목적지 포트 \| 순서번호 \| [실제 데이터]` |
| 인터넷층 | 패킷 | `출발지 IP \| 목적지 IP \| TTL \| 프로토콜 \| [TCP 세그먼트]` |
| 데이터링크층 | 프레임 | `목적지 MAC \| 출발지 MAC \| 타입 \| [IP 패킷] \| FCS` |

> - **FCS (Frame Check Sequence)**: 이더넷 프레임 끝에 붙는 에러 검증값
> - **TTL (Time To Live)**: 패킷이 거칠 수 있는 최대 홉 수. 0이 되면 패킷 폐기 (무한 순환 방지)
> - 송신 시 위→아래(세그먼트->패킷->데이터 프레임)로 헤더 추가 (캡슐화), 수신 시 아래→위로 헤더 제거 (역캡슐화)

### 이더넷 (Ethernet)
- LAN에서 가장 많이 쓰이는 유선 통신 표준
- 데이터링크층 + 물리층 담당
- MAC 주소 사용, 전송 단위는 프레임(Frame)

| | 이더넷 | 와이파이 |
|---|---|---|
| 연결 | 유선 (케이블) | 무선 |
| 속도 | 빠르고 안정적 | 상대적으로 느리고 불안정 |
| 보안 | 강함 (물리 접근 필요) | 약함 (무선 도청 가능) |

**이더넷 보안**

| 구분 | 내용 |
|---|---|
| **강점** | 케이블 직접 연결 없이 접속 불가 + 전파 외부 유출 없음 / 스위치가 목적지 포트에만 패킷 전달 → 불필요한 경유 없음 |
| **취약점** | 내부망 침투 시 ARP 스푸핑 취약 → 물리 포트 관리 필요 |
| **필터링** | MAC 필터링 / VLAN / 포트 보안으로 내부망 접근 제어 가능 |

| 개념 | 설명 |
|---|---|
| **MAC 필터링** | 허용된 MAC 주소만 통신 가능하도록 스위치에서 설정 |
| **VLAN** | 같은 물리 네트워크를 논리적으로 분리 → 부서별 트래픽 격리 |
| **포트 보안** | 특정 포트에 연결 가능한 MAC 수 제한 |
| **ARP 스푸핑** | ARP는 IP→MAC 변환 프로토콜. 공격자가 거짓 ARP 응답으로 라우터인 척 등록 → 트래픽 경유 (MITM). 방어: 정적 ARP 고정, HTTPS/VPN |
### MAC vs IP
| | MAC | IP |
|---|---|---|
| **구조** | 랜덤 (제조사 기반, 위치 무관) | 체계적 (계층적 주소, 위치 반영) |
| **변경** | 고정 (하드웨어) | 유동 (네트워크마다 바뀜) |
| **유리한 경우** | LAN 내 장치 식별, 기기 인증, MAC 필터링 | 라우터 경로 탐색, 전 세계 목적지 식별 |
| **비유** | 주민등록번호 (고유하나 위치 정보 없음) | 집 주소 (위치 반영, 체계적) |

> MAC만으로는 전국 배달 불가 → IP와 함께 써야 함

### Switch vs. Router
| | Switch | Router |
|---|---|---|
| **개념** | 같은 네트워크(LAN) 내 장치 연결, MAC 주소로 해당 장치에만 전달 | 서로 다른 네트워크 간 연결, IP 주소로 최적 경로 탐색 후 전달 |
| **계층** | 물리/링크 (2계층) | 물리/링크/네트워크 (3계층) |
| **주소** | 물리 주소 (MAC) | 논리 주소 (IP) |
| **규모** | 소규모, 저가 | 중대형, 고가 |
| **용도** | 사용자 연결 | 외부망 연결 |
| **예시** | 회사 내부 PC 연결 | 가정용 공유기, 인터넷 연결 |

> - **비유**: 스위치 = 같은 아파트 동 안에서 호수 찾아주는 것 / 라우터 = 다른 도시로 보낼 때 경로 찾아주는 것

## 실 전송 과정

| 단계 | 장치 | 동작 |
|---|---|---|
| 1 | **PC A** | 데이터를 패킷으로 분할. 목적지 IP 확인 → 외부망이므로 라우터로 전송 결정. ARP로 라우터 MAC 확인 후 이더넷 프레임 생성 |
| 2 | **스위치** | 프레임의 목적지 MAC 확인 → MAC 테이블 조회 → 라우터 포트로만 전달 (이더넷) |
| 3 | **라우터** | 프레임 수신 후 IP 패킷 꺼냄 → 목적지 IP로 라우팅 테이블 조회 → 최적 경로 결정 → MAC 주소를 다음 홉으로 교체 후 새 프레임 생성 |
| 4 | **인터넷 구간** | 라우터를 여러 번 거치며 MAC은 홉마다 교체, IP는 유지 |
| 5 | **수신측 스위치** | 목적지 MAC 확인 → PC B 포트로 전달 |
| 6 | **PC B** | 프레임 수신 → IP 패킷 → 포트 번호로 해당 프로세스(앱)에 전달 |

> - **IP는 출발지부터 목적지까지 유지**, **MAC은 홉마다 교체**
> - 이더넷은 LAN 구간(PC↔스위치↔라우터)에서 동작, 라우터 밖은 다른 프로토콜로 전환될 수 있음
> - **MAC이 홉마다 교체되는 이유**: MAC은 같은 LAN 안에서만 유효한 주소(데이터 링크 계층에선 같은 LAN선에 있지 않는 한 상대 MAC주소를 찾지 못해 그 경로의 라우터 주소를 대신 저장). 라우터를 넘으면 새로운 네트워크(LAN)가 시작되므로 해당 구간의 MAC으로 교체 필요. IP는 전체 경로의 목적지를 나타내므로 유지(IP가 2계층의 MAC주소를 정해주는 주체가됨: IP:목적지  MAC:정거장, 톨게이트 역할)

### 상세 예시
> PC A (192.168.0.10) → 웹서버 B (123.45.67.89:80) 로 HTTP 요청 전송

**① PC A에서 출발**
- 목적지 IP `123.45.67.89` 확인 → 다른 네트워크이므로 기본 게이트웨이(라우터)로 전송
- ARP: "라우터(192.168.0.1)의 MAC이 뭐야?" → 라우터 응답: `AA:BB:CC:DD:EE:01`
- 이더넷 프레임 생성:
  - 출발지 MAC: `11:22:33:44:55:66` (PC A)
  - 목적지 MAC: `AA:BB:CC:DD:EE:01` (라우터)
  - 출발지 IP: `192.168.0.10`
  - 목적지 IP: `123.45.67.89`
  - 포트: 출발 `54321` → 목적지 `80` (HTTP)

**② 스위치 통과**
- MAC 테이블에서 `AA:BB:CC:DD:EE:01` 확인 → 라우터 연결 포트로만 전달
- IP/포트는 보지 않음

**③ 라우터 처리**
- 프레임에서 IP 패킷 꺼냄
- 라우팅 테이블에서 `123.45.67.89` 경로 탐색 → 다음 홉 결정
- MAC 교체: 목적지 MAC을 다음 홉 라우터 MAC으로 변경
- 새 이더넷 프레임으로 재포장 후 전송

**④ 인터넷 구간**
- 여러 라우터를 거치며 반복
- IP(`192.168.0.10` → `123.45.67.89`)는 유지
- MAC은 매 홉마다 교체

**⑤ 수신측 도착**
- 웹서버 측 스위치: MAC 확인 → 웹서버 포트로 전달
- 웹서버 B: 포트 `80` 확인 → HTTP 프로세스로 전달 → 응답 생성
