# 제 11장 데이터링크 제어(DLC)

---

## 데이터 링크층의 두 계층 구조

데이터 링크층은 두 개의 서브계층으로 나뉜다.

```
┌─────────────────────────────────────┐
│   Data Link Control (DLC) Sublayer  │  ← 위 층
├─────────────────────────────────────┤
│ Medium Access Control (MAC) Sublayer│  ← 아래층
└─────────────────────────────────────┘
```

- **위 층 : 데이터 링크 제어(DLC)** — 이 장에서 다루는 내용
- **아래층 : 매체 접근 제어(MAC)** — 공유하는 매체에 대한 접근 제어 책임

---

## 11.1 DLC 서비스

### 프레임 짜기(Framing)

데이터를 프레임 단위로 묶어 전송하는 방법.

#### 1. 고정 크기 프레임(Fixed-size Frame)
- 프레임의 길이가 고정됨
- 예: **ATM 광역 네트워크의 셀(Cell)**
- 프레임의 경계를 크기 자체로 알 수 있어 별도의 구분자 불필요

#### 2. 가변 크기 프레임(Variable-size Frame)
- 주로 **LAN에서 사용**
- 프레임이 끝나는 곳과 다음 프레임이 시작하는 곳을 **명시적으로 지정**해야 함
- 두 가지 방식으로 나뉨:

---

### 2-1. 문자(바이트) 중심 프로토콜 (Character-oriented Protocol)

- 전달되는 데이터는 **코딩 시스템의 8비트 문자** (ASCII 등)
- 프레임의 **시작과 마지막에 플래그(Flag) 추가**

#### 프레임 구조

```
┌──────┬────────┬──────────────────────────┬─────────┬──────┐
│ Flag │ Header │  Data (가변 길이 문자들)   │ Trailer │ Flag │
└──────┴────────┴──────────────────────────┴─────────┴──────┘
         ←────── Data from upper layer ──────→
              Variable number of characters
```

---

#### 문자 채워 넣기(Stuffing)와 빼기(Unstuffing)

**문제:** 정보 내에 플래그로 사용되는 패턴이 있을 경우, 수신측이 이를 프레임 끝으로 **오인**할 수 있음.

**해결:** ESC(Escape) 문자를 삽입하여 데이터 내의 Flag/ESC를 구분

**규칙:**
- 데이터 내에 **Flag 문자**가 있으면 → 앞에 **ESC 삽입**
- 데이터 내에 **ESC 문자**가 있으면 → 앞에 **ESC 삽입** (ESC ESC)

```
[원본 데이터]
┌────────┬──────┐
│  Flag  │ ESC  │
└────────┴──────┘
        ↓ Stuffed (ESC 삽입)

[Frame sent]
┌──────┬────────┬─────┬──────┬─────┬─────┬─────────┬──────┐
│ Flag │ Header │ ESC │ Flag │ ... │ ESC │ Trailer │ Flag │
└──────┴────────┴─────┴──────┴─────┴─────┴─────────┴──────┘
                  ↑ Extra 2 bytes (ESC 2개 추가)

[Frame received]
┌──────┬────────┬─────┬──────┬─────┬─────┬─────────┬──────┐
│ Flag │ Header │ ESC │ Flag │ ... │ ESC │ Trailer │ Flag │
└──────┴────────┴─────┴──────┴─────┴─────┴─────────┴──────┘
        ↓ Unstuffed (ESC 제거)

[복원된 데이터]
┌────────┬──────┐
│  Flag  │ ESC  │
└────────┴──────┘
```

---

### 2-2. 비트 중심 프로토콜 (Bit-oriented Protocol)

- 프레임의 데이터 부분을 전부 **비트 열(bit stream)** 로 인식
- 플래그는 **`01111110`** 비트 패턴 사용

#### 프레임 구조

```
┌──────────┬────────┬──────────────────────────────┬─────────┬──────────┐
│ 01111110 │ Header │  01111010110 ··· 11011110     │ Trailer │ 01111110 │
└──────────┴────────┴──────────────────────────────┴─────────┴──────────┘
   Flag                  Variable number of bits                  Flag
```

---

#### 비트 채우기(Bit Stuffing)와 빼기(Bit Unstuffing)

**문제:** 데이터 내에 `01111110` (6개의 연속 1) 패턴이 있으면 플래그로 오인

**해결:** 데이터 내에서 **연속으로 1이 5개** 나오면 그 뒤에 **0을 강제 삽입(stuff)**

```
[원본 데이터]
0001111111100111110 1000
         (연속 1이 6개)   (연속 1이 5개)

[Stuffed → Frame sent]
0001111101100111110 01000
         ↑ Extra 0    ↑ Extra 0
         (5개 1 뒤에 0 삽입)

[Frame received → Unstuffed]
0001111111100111110 1000  ← 원본 복원
```

> **stuffing 규칙 요약:**
> - 송신: 연속 1이 5개 나오면 → 뒤에 0 삽입
> - 수신: 연속 1이 5개 후 0을 만나면 → 그 0을 제거 (unstuff)
> - 수신: 연속 1이 5개 후 1을 만나면 → 플래그 (`01111110`)로 인식

## 11.1.2 흐름 제어와 오류 제어 (Flow and Error Control)

### 1. 흐름 제어(Flow Control)
- 송신자가 **확인응답(ACK)을 받기 전**에 보낼 수 있는 **데이터의 양을 제한**하기 위해 사용하는 일련의 절차
- 수신자가 처리 못할 속도로 데이터가 쏟아지는 것을 방지

### 2. 오류 제어(Error Control)
- **데이터 재전송을 요구하는 ARQ(Automatic Repeat Request)** 를 기반으로 함
- 오류 발생 시 해당 프레임을 재전송 요청

---

## 11.2 데이터링크 계층 프로토콜

흐름과 오류 제어를 다루는 데이터링크 계층의 프로토콜 4가지:

| 번호 | 프로토콜 | 특징 |
|------|---------|------|
| 1 | **단순 프로토콜** | 흐름/오류 제어 없음 |
| 2 | **정지/대기 프로토콜** | 1개 전송 후 ACK 대기 |
| 3 | **Go-Back-N 프로토콜** | 윈도우 방식, 오류 시 N번째부터 재전송 |
| 4 | **선택적-반복 프로토콜** | 윈도우 방식, 오류 난 것만 재전송 |

---

### 11.2.2 정지/대기 프로토콜 (Stop-and-Wait Protocol)

프레임 하나를 보내고 ACK를 받을 때까지 **정지하고 대기**하는 방식.

```
[Sending node]                              [Receiving node]
   Network                                     Network
      │                                            │
   Data-link ──── Frame (+ CRC) ─────────→ Data-link
      │                                            │
   Data-link ←─── ACK (+ CRC) ─────────── Data-link
      │                                            │
   ⏱ Timer   ←────── Logical link (duplex) ──────→
```

**동작 원리:**
- 송신측: 프레임 전송 후 **타이머(Timer) 시작**, ACK 대기
- 수신측: 프레임 수신 후 **CRC 오류 검사** → 정상이면 ACK 전송
- 타이머 만료 전 ACK 미수신 → **재전송**
- 각 프레임에 **CRC(Cyclic Redundancy Check)** 포함하여 오류 검출

**단점:** 한 번에 하나의 프레임만 전송 → **링크 효율이 낮음**

## 11.4 Point-to-Point Protocol (PPP)

### PPP란?

**점대점 액세스(Point-to-Point Access)에 사용되는 문자 중심 프로토콜**

- PC와 워크스테이션 간에 **모뎀이나 케이블을 연결**하여 정보를 주고받는 데 사용
- 두 대의 컴퓨터가 **직렬 인터페이스**를 이용하여 통신할 때 데이터를 송수신하는 데 사용하는 프로토콜
- 특히 **전화회선을 통해 서버에 연결하는 PC**에서 자주 사용됨

---

### PPP 프레임 구조 (Figure 11.32)

```
┌──────┬─────────┬─────────┬──────────────┬──────────┬─────────────┬──────┐
│ Flag │ Address │ Control │   Protocol   │ Payload  │     FCS     │ Flag │
├──────┼─────────┼─────────┼──────────────┼──────────┼─────────────┼──────┤
│1 byte│ 1 byte  │ 1 byte  │ 1 or 2 bytes │ Variable │ 2 or 4 bytes│1 byte│
└──────┴─────────┴─────────┴──────────────┴──────────┴─────────────┴──────┘
  11111111         11000000
```

| 필드 | 크기 | 설명 |
|------|------|------|
| **Flag** | 1 byte | 프레임 시작/끝 구분자 (`11111111`) |
| **Address** | 1 byte | 항상 `11111111` (브로드캐스트) |
| **Control** | 1 byte | 항상 `11000000` |
| **Protocol** | 1 or 2 bytes | 페이로드의 프로토콜 종류 지정 |
| **Payload** | 가변 | 실제 전송 데이터 |
| **FCS** | 2 or 4 bytes | 오류 검출용 체크섬(Frame Check Sequence) |
| **Flag** | 1 byte | 프레임 끝 |

---

### PPP의 다중화 (Figure 11.34 — Multiplexing in PPP)

PPP는 **Protocol 필드**를 통해 여러 계층 프로토콜을 하나의 프레임으로 다중화한다.

```
[Network layer]
  Data from different networking protocols ──→ Data

[Data link layer]
  ┌─────────────────────────────────────────────────────┐
  │  NCP (Network Control Protocol)                     │
  │    ├── IPCP (IP Control Protocol)                   │
  │    └── OSI CP                                       │
  │  AP (Authentication Protocol)                       │
  │    ├── PAP  (Password Authentication Protocol)      │
  │    └── CHAP (Challenge Handshake Auth Protocol)     │
  │  LCP (Link Control Protocol)                        │
  └─────────────────────────────────────────────────────┘
         ↓ Protocol 필드 값으로 구분
```

#### Protocol 필드 값

| 값 | 프로토콜 |
|----|---------|
| `0xC021` | LCP (Link Control Protocol) |
| `0xC023` | PAP (Password Authentication Protocol) |
| `0xC223` | CHAP (Challenge Handshake Authentication Protocol) |
| `0x8021` 이상 | NCP (Network Control Protocol) |
| `0x0021` 이상 | Data (IP 등) |

**각 프로토콜 약어:**
- **LCP** : Link Control Protocol — 링크 설정, 유지, 해제 담당
- **AP** : Authentication Protocol — 인증 담당 (PAP, CHAP)
- **NCP** : Network Control Protocol — 네트워크 계층 프로토콜 설정 담당

---

### PPP 전환 단계 (Transition Phases, Figure 11.33)

PPP 연결의 상태 전이 과정:

```
         Failed
            ↓
         Carrier detected
[Dead] ──────────────────→ [Establish]
  ↑                              │
  │ Carrier dropped              │ Options agreed by both sides
  │                              ↓
[Terminate] ←── Failed ── [Authenticate] ─── (If authentication not needed)
     ↑                          │                        │
     │ Done                     │ Authentication         │
     │               successful ↓                        │
  [Open] ←── Network layer ── [Network] ←───────────────┘
             configuration
```

| 상태 | 설명 |
|------|------|
| **Dead** | 물리 계층 링크 없음 (초기 상태) |
| **Establish** | 캐리어 감지 → LCP를 이용해 옵션 협상 |
| **Authenticate** | PAP 또는 CHAP으로 인증 수행 |
| **Network** | NCP를 이용해 네트워크 계층 설정 |
| **Open** | 데이터 전송 가능 상태 (정상 통신) |
| **Terminate** | 연결 종료 중 |

## 인증 프로토콜

### PAP (Password Authentication Protocol)

**2단계 처리절차를 가진 간단한 인증 프로토콜**

```
[User]                                         [System]
  │──── Authenticate-request ─────────────────→│
  │     (사용자이름 + 패스워드 평문 전송)          │
  │                                             │
  │←─── Authenticate-ack / authenticate-nak ──│
  │     (수용 또는 거부)                          │
```

**단계:**
1. 사용자이름과 패스워드를 **평문(plaintext)** 으로 보낸다
2. 시스템이 연결을 **수용(ack)** 하거나 **거부(nak)** 한다

**PAP 패킷 구조:**

| 패킷 종류 | Code | 필드 구성 |
|-----------|------|-----------|
| Authenticate-request | Code: 1 | ID \| Length \| User name length \| **User name** \| Password length \| **Password** |
| Authenticate-ack | Code: 2 | ID \| Length \| Message length \| **User name** |
| Authenticate-nak | Code: 3 | ID \| Length \| Message length \| **User name** |

PPP 프레임 캡슐화:
```
┌──────┬─────────┬─────────┬────────┬──────────────────┬─────┬──────┐
│ Flag │ Address │ Control │ C023₁₆│ Payload(+padding) │ FCS │ Flag │
└──────┴─────────┴─────────┴────────┴──────────────────┴─────┴──────┘
                             ↑ PAP 프로토콜 식별자
```

---

### CHAP (Challenge Handshake Authentication Protocol)

**높은 보안을 제공하는 3방향 핸드세이크(3-way handshake) 인증 프로토콜**

> PAP과 달리 패스워드를 직접 전송하지 않아 보안성이 높음

```
[User]                                              [System]
  │←─── Challenge (챌린지 패킷 전송) ──────────────│
  │                                                  │
  │──── Response (함수 적용 결과 전송) ─────────────→│
  │                                                  │ (같은 함수 적용 후 비교)
  │←─── Success or Failure ────────────────────────│
```

**단계:**
1. **시스템**이 챌린지(Challenge) 패킷을 사용자에게 보냄
2. **사용자**는 미리 정의된 함수(해시 함수)를 챌린지에 적용하여 결과값을 만들고 시스템에 전송
3. **시스템**은 같은 함수를 적용하여 결과값을 만들고, 사용자가 보낸 값과 **비교**하여 일치하면 허용

**CHAP 패킷 구조:**

| 패킷 종류 | Code | 필드 구성 |
|-----------|------|-----------|
| Challenge | Code: 1 | ID \| Length \| Challenge length \| **Challenge value** \| Name |
| Response  | Code: 2 | ID \| Length \| Response length \| **Response value** \| Name |
| Success   | Code: 3 | ID \| Length \| Message |
| Failure   | Code: 4 | ID \| Length \| Message |

PPP 프레임 캡슐화:
```
┌──────┬─────────┬─────────┬────────┬──────────────────┬─────┬──────┐
│ Flag │ Address │ Control │ 0xC223 │ Payload(+padding) │ FCS │ Flag │
└──────┴─────────┴─────────┴────────┴──────────────────┴─────┴──────┘
                              ↑ CHAP 프로토콜 식별자
```

## 핵심 정리

| 개념 | 요약 |
|------|------|
| **DLC** | 데이터 링크층 위 서브계층, 프레임 짜기/흐름제어/오류제어 담당 |
| **고정 프레임** | ATM 셀, 크기 고정 |
| **가변 프레임** | LAN, 문자 중심 or 비트 중심 |
| **문자 stuffing** | ESC를 이용해 Flag/ESC 문자를 데이터 내에서 구분 |
| **비트 stuffing** | 연속 1이 5개면 0 삽입, 수신 시 제거 |
| **흐름 제어** | ACK 전에 보낼 수 있는 데이터 양 제한 |
| **오류 제어** | ARQ 기반 재전송 |
| **정지/대기** | 1프레임 전송 → ACK 대기 → 다음 전송 |
| **PPP** | 점대점 연결용 문자 중심 프로토콜, 전화회선에 주로 사용 |
| **PPP 필드** | Flag / Address / Control / Protocol / Payload / FCS / Flag |
| **LCP** | PPP 링크 설정·유지·해제 |
| **NCP** | 네트워크 계층 프로토콜 설정 |
| **PAP** | 2단계 인증, 패스워드 평문 전송 (보안 취약) |
| **CHAP** | 3단계 인증, 챌린지-응답 방식 (보안 강함) |
