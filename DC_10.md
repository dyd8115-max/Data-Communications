## 데이터 링크층의 두 계층 구조

데이터 링크층은 두 개의 서브계층으로 나뉨

```
┌─────────────────────────────────────┐
│   Data Link Control (DLC) Sublayer  │  ← 위 층 : 데이터 링크 제어
├─────────────────────────────────────┤
│ Medium Access Control (MAC) Sublayer│  ← 아래층 : 공유하는 매체에 대한 접근 제어 책임
└─────────────────────────────────────┘
```
>이번 장은 아래층 중심 설명

## 다중 접속 프로토콜

### 다중접속 (Multiple Access)
- 노드가 멀티포인트나 공통 링크를 사용할 때 링크에 대한 접속을 조절할 수 있는 다중접속 프로토콜이 필요

```
다중접속 프로토콜
├── 임의접속 (Random Access)
│   ├── ALOHA
│   ├── CSMA
│   ├── CSMA/CD
│   └── CSMA/CA
├── 제어접속 (Controlled Access)
│   ├── Reservation (예약)
│   ├── Polling (폴링)
│   └── Token passing (토큰전달)
└── 채널화 (Channelization)
    ├── FDMA
    ├── TDMA
    └── CDMA
```

---

## 12.1 임의접속 (Random Access)

### 개요
- 각 지국은 다른 어느 지국에 의해 제어받지 않은 매체 접근 권리를 가짐
- 충돌을 피하기 위한 절차 필요

**충돌 회피를 위해 답해야 할 4가지 질문:**
1. 언제 노드가 매체에 접속할 수 있는가?
2. 만약 매체가 사용된다면 노드는 무엇을 할 수 있는가?
3. 어떤 방법으로 노드는 전송의 실패와 성공을 파악할 수 있는가?
4. 만약 매체 충돌이 발생한다면 노드는 무엇을 할 수 있는가?

### 임의접속 방법의 진화
```
MA → CSMA → CSMA/CD
              └→ CSMA/CA
```

## 12.1.1 ALOHA

### 다중접속 (MA)
- 최초의 다중접속 방법
- 9,600bps를 가진 무선 LAN에 사용되도록 설계

---

### 순수 ALOHA (Pure ALOHA)

- 전송할 데이터가 있으면 **즉시 전송**
- 충돌 발생 시 임의 시간 대기 후 재전송
- 충돌 감지 없음 → 확인응답(ACK)으로 성공/실패 판단

#### Figure 12.3 — 순수 ALOHA 네트워크에서 프레임
```
Station 1  ──[Frame 1.1]────────────────[Frame 1.2]──────→ Time
Station 2  ──────[Frame 2.1]──────────────[Frame 2.2]────→ Time
Station 3  ────────[Frame 3.1]──────────────[Frame 3.2]──→ Time
Station 4  ──────────[Frame 4.1]────────────[Frame 4.2]──→ Time
                     ↑ Collision            ↑ Collision
                     duration               duration
```

#### Figure 12.5 — Vulnerable time (순수 ALOHA)
```
     B's end      A's end
     collides     collides
     with A's     with C's
     beginning    beginning
         ↓           ↓
 ├──[B]──┤           ├──────────[A]──────────┤  ├──[C]──┤
 t - T_fr           t                        t + T_fr
 ←──────────── Vulnerable time = 2 × T_fr ──────────────→
```
> Vulnerable time = **2 × T_fr** (프레임 전송시간의 2배)
> 이 구간 안에 다른 지국이 전송을 시작하면 충돌 발생

#### 처리율 (Throughput)

$$S = G \times e^{-2G}$$

- **G**: 한 프레임 전송시간에 시스템에서 생성되는 프레임의 평균 개수
- **최대 처리율**: $S_{max} = 0.184$ when $G = \frac{1}{2}$
  - 즉, 채널 용량의 최대 **18.4%** 만 활용 가능

#### Example 12.3

> 순수 ALOHA 네트워크가 200kbps 공유 채널에서 200비트 프레임을 전송한다.
> 시스템이 초당 다음과 같이 프레임을 생성할 때 처리율은?
> a. 1000 frames/s   b. 500 frames/s   c. 250 frames/s

**Solution:**
- 프레임 전송 시간: 200bit ÷ 200kbps = **1ms**

| 경우 | G (frames/ms) | S = G × e^(-2G) | 처리율 |
|------|--------------|-----------------|--------|
| a. 1000 frames/s | 1 | 0.135 (13.5%) | 135 frames/s |
| b. 500 frames/s | 0.5 | 0.184 (18.4%) | 92 frames/s |
| c. 250 frames/s | 0.25 | 0.152 (15.2%) | 38 frames/s |

> a의 경우: 1000개 중 약 135개만 성공적으로 전달됨

---

### 슬롯 ALOHA (Slotted ALOHA)

- 시간을 **슬롯(slot)** 단위로 나눔 (슬롯 크기 = 프레임 전송 시간 T_fr)
- 지국은 슬롯의 **시작점에서만** 전송 가능
- 충돌 가능 구간이 순수 ALOHA의 절반으로 줄어듦

#### Figure 12.6 — Frames in a slotted ALOHA network
```
Station 1  ──[Frame 1.1]──────────[Frame 1.2]────────────→ Time
Station 2  ──────[Frame 2.1]────────[Frame 2.2]──────────→ Time
Station 3  ──────[Frame 3.1]──────────[Frame 3.2]────────→ Time
Station 4  ──────[Frame 4.1]──────────[Frame 4.2]────────→ Time
           Slot1  Slot2  Slot3  Slot4  Slot5  Slot6
```

#### Figure 12.7 — Vulnerable time (슬롯 ALOHA)
```
 A collides with C
         ↓
 ├────────────[B]────────────┤  ├────────────[A]────────────┤  ├─────[C]─────┤
                             t                               t + T_fr
                             ←── Vulnerable time = T_fr ────→
```
> Vulnerable time = **T_fr** (순수 ALOHA의 절반)

#### 처리율

$$S = G \times e^{-G}$$

- **최대 처리율**: $S_{max} = 0.368$ when $G = 1$
  - 채널 용량의 최대 **36.8%** 활용 가능 (순수 ALOHA의 2배)

#### Example 12.4

> 슬롯 ALOHA 네트워크, 동일 조건 (200kbps, 200비트 프레임)

| 경우 | G | S = G × e^(-G) | 처리율 |
|------|---|----------------|--------|
| a. 1000 frames/s | 1 | 0.368 (36.8%) | 368 frames/s |
| b. 500 frames/s | 0.5 | 0.303 (30.3%) | 152 frames/s |
| c. 250 frames/s | 0.25 | 0.195 (19.5%) | 49 frames/s |

#### 순수 ALOHA vs 슬롯 ALOHA 비교

| | 순수 ALOHA | 슬롯 ALOHA |
|---|---|---|
| 전송 시점 | 언제든지 | 슬롯 시작점에서만 |
| Vulnerable time | 2 × T_fr | T_fr |
| 최대 처리율 | 18.4% | 36.8% |
| 동기화 필요 | 불필요 | 필요 |

## 12.1.2 CSMA (반송파 감지 다중접속)

### 개요
- **Carrier Sense Multiple Access**
- 충돌 가능성을 줄이기 위해 개발
- 각 지국은 **전송 전 매체의 상태를 점검**
- 충돌 가능성을 줄일 수는 있지만 **완전히 제거는 불가**
  - 전파지연 때문에 충돌 가능성은 항상 존재

---

### 지속성 전략 (Persistence Strategy)

매체가 사용 중일 때 단말의 진행 절차를 정의

#### 비지속성 전략 (Nonpersistent Strategy)
- 회선을 감지해서 사용 중이 아니면 **즉시 전송**
- 사용 중이면 **임의의 시간 동안 대기**하다 다시 회선 감지
- 장점: 충돌 가능성 낮음
- 단점: 회선이 비어있어도 대기하는 경우 발생 → 효율 저하

#### 지속성 전략 (Persistent Strategy)
- 회선을 감지해서 사용 중이 아니면 즉시 전송
- 사용 중이면 **계속 감지하다가** 비면 전송

| 방식 | 설명 |
|------|------|
| **1-지속성** | 회선이 비면 확률 1로 즉시 전송 (항상 전송) |
| **P-지속성** | 회선이 비면 p의 확률로 전송, (1-p)의 확률로 대기 |

> **P-지속성 예시**: P = 0.2이면 회선이 비었을 때 20% 확률로 전송, 80% 확률로 전송 중단

#### Figure 12.10 — 세 가지 지속성 방법의 동작
```
1-지속성:
  감지 → 사용중? → NO → 즉시 전송
                → YES → 계속 감지 → 비면 즉시 전송 (충돌 빈번)

비지속성:
  감지 → 사용중? → NO → 즉시 전송
                → YES → 임의 시간 대기 → 다시 감지

P-지속성:
  감지 → 사용중? → NO → p 확률로 전송 / (1-p) 확률로 다음 슬롯 대기
                → YES → 계속 감지
```

#### Figure 12.11 — 세 가지 지속성 방법의 흐름도
```
[비지속성]                    [1-지속성]               [P-지속성]
    ↓                             ↓                        ↓
Sense carrier              Sense carrier             Sense carrier
    ↓                             ↓                        ↓
  Busy? → Yes → Wait         Busy? → Yes → Wait         Busy? → Yes → Wait
    ↓ No                       ↓ No                       ↓ No
Send the frame             Send the frame          Use back-off process
                                                   as though collision occurred
                                                   Station can transmit
```

## 12.1.3 CSMA/CD (충돌 검출)

**Carrier Sense Multiple Access with Collision Detection**

- 충돌을 **처리하는 절차**를 추가
- 충돌 발생 시 재전송 요구
- 두 번째 충돌을 줄이기 위해 대기

### 백오프 (Back-off) 방법
지속적인 백오프 방법에서 대기 시간:
$$0 \sim 2^N \times \text{최대전송시간}$$
- N: 전송 시도 횟수

### Figure 12.12 — CSMA/CD에서 첫 번째 비트의 충돌
```
A         B             C             D
│         │             │             │
├→ First bit of A ─────────────────→  │
│         │             │         T_p │ Transmission
│  T_AB   │ A's         │             │ time
├─────────┤ collision   │             │
│         │ detection & │             │
│         │ abort       │         C's │
│         │             │    collision│
│         │      First bit of C ──────┤
│         │             │         abort
T_0                                  T_0 + 2T_p
```
> 최악의 경우 충돌 감지까지 최대 **2 × T_p** 시간 필요 (T_p: 전파 지연 시간)

### Example 12.5

> CSMA/CD 네트워크, 대역폭 10Mbps, 최대 전파지연 25.6μs
> → 최소 프레임 크기는?

**Solution:**
- 프레임 전송 시간: $T_{fr} = 2 \times T_p = 51.2\mu s$
- 최소 프레임 크기: $10Mbps \times 51.2\mu s = 512bits = 64bytes$

> 이것이 표준 이더넷의 최소 프레임 크기 **(64bytes)**

### Figure 12.14 — CSMA/CD 흐름도
```
Start
  ↓
Frame ready to send
  ↓
Sense carrier → Busy? → YES → [지속성 전략에 따라 대기]
  ↓ NO
Transmit & monitor for collision
  ↓
Collision detected?
  ↓ YES                    ↓ NO
Send jam signal         Transmission successful
  ↓
Increment attempt counter
  ↓
Max attempts exceeded? → YES → Abort (전송 포기)
  ↓ NO
Wait back-off time
  ↓
Try again
```

## 12.1.4 CSMA/CA (충돌 회피)

**Carrier Sense Multiple Access with Collision Avoidance**

### 개요
- **무선 네트워크에서 사용** (Wi-Fi 등)
- 무선은 충돌을 감지하지 못함 → 충돌 **회피** 방식 사용
- **숨겨진 지국 문제(Hidden Station Problem)** 해결
  - A와 C가 서로 감지하지 못한 채 B에게 동시 전송하는 문제

### 구성 요소
- **프레임간 공간 (IFS)**: 채널이 비어도 바로 전송하지 않고 일정 시간 대기
- **경쟁 구간 (Contention Window)**: 임의 슬롯을 선택해 대기
- **확인응답 (ACK)**: 수신측이 성공 수신을 알림

### RTS-CTS-DATA-ACK 절차
```
송신측                              수신측
  │──── RTS (Request To Send) ────→│
  │                                 │
  │←─── CTS (Clear To Send) ───────│
  │                                 │
  │──── DATA ───────────────────────→│
  │                                 │
  │←─── ACK ────────────────────────│
```
> RTS/CTS 교환으로 숨겨진 지국 문제 해결
> 다른 지국들은 CTS를 수신하면 일정 시간 전송 자제

### Figure 12.15 — CSMA/CA 흐름도
```
Start
  ↓
Frame ready
  ↓
Wait IFS time
  ↓
Channel idle?
  ↓ YES                      ↓ NO
Send RTS                  Wait + back-off
  ↓
Receive CTS?
  ↓ YES       ↓ NO
Send DATA   Back-off & retry
  ↓
ACK received?
  ↓ YES    ↓ NO
Success   Retry
```

## 12.2 제어 접속 (Controlled Access)

### 개요
- 어느 지국이 송신 권한을 가지고 있는지 **서로 협력**하여 결정
- 충돌 발생 없음

### 종류
1. **예약 (Reservation)**
2. **폴링 (Polling)**
3. **토큰 전달 (Token Passing)**

---

### 예약 (Reservation)

- 지국은 데이터를 송신하기 전에 **예약**이 필요
- N개의 지국이 존재하면 **N개의 예약된 미니 슬롯(mini slot)** 들이 예약 프레임 안에 존재
- 예약을 한 지국은 데이터 프레임을 예약 프레임 뒤에 전송

#### Figure 12.18 — Reservation access method
```
예약 프레임:
┌──┬──┬──┬──┬──┐
│1 │2 │3 │4 │5 │  ← N개의 미니 슬롯 (각 지국의 예약용)
└──┴──┴──┴──┴──┘
     ↓ 예약 완료 후
┌──────────┬──────────┬──────────┬──────────┐
│ Data     │ Data     │ Data     │ Data     │
│ station1 │ station4 │ station3 │ station1 │
└──────────┴──────────┴──────────┴──────────┘
```

---

### 폴링 (Polling)

- **주국(Master)** 과 **종국(Slave)** 으로 구성된 토폴로지에서 동작

#### 폴 (Poll)
- 주국이 **데이터 수신**을 원할 때 종국 장치에게 문의
- 주국이 종국으로부터 전송을 요청하는 데 사용

#### 선택 (Selection)
- 주국이 **데이터 송신**을 원할 때 종국장치에게 알려줌
- 주국이 언제든지 송신할 것이 있을 때 사용
- 예정된 전송을 위해 주국은 종국의 **준비 상태에 대한 확인 응답**을 대기
- 주국은 전송 예정된 장치의 주소를 한 필드에 포함하고 **선택 프레임(SEL)** 을 만들어 전송

#### Figure 12.19 — Select and poll functions in polling access method
```
[Selection 과정]             [Poll 과정]
Primary                      Primary
  │                             │
  │──── SEL ────→ A             │──── Poll ────→ A
  │←─── ACK ────             A │←─── NAK ─────  (전송할 데이터 없음)
  │──── Data ──→ A             │──── Poll ────→ B
  │←─── ACK ────             B │←─── Data ────  (데이터 전송)
                               │──── ACK ─────→ B
```

---

### 토큰 전달 (Token Passing)

- **토큰(Token)** 을 가진 지국이 데이터 송신 권한을 가짐
- 토큰은 네트워크를 순환하며 지국 간에 전달됨

#### 토큰 전달 네트워크
```
A ────→ B ────→ C ────→ D
↑                        │
└────────────────────────┘
        (토큰 순환)
```
- 토큰 보유 지국이 전송 후 다음 지국에 토큰 전달
- 전송할 데이터가 없으면 즉시 토큰 전달

---

## 12.3 채널화 (Channelization)

채널의 **대역폭을 시간, 주파수, 코드**로 나눠 여러 지국이 동시에 사용할 수 있도록 하는 방법

---

### FDMA (주파수 분할 다중접속)

**Frequency Division Multiple Access**

- 사용 가능한 대역폭은 **모든 지국들에 의해 공유**
- 각 지국들은 **할당된 대역**을 사용하여 데이터를 전송
- 각각의 대역은 **특정 지국을 위해 예약**되어 있음

```
주파수 대역:
┌────┬────┬────┬────┬────┐
│ 지국1│ 지국2│ 지국3│ 지국4│ ...│  ← 각 지국에 고정 주파수 대역 할당
└────┴────┴────┴────┴────┘
          ↓ Common channel
```

> 6장 FDM과 개념 동일. 차이는 FDM은 물리층 다중화 기술, FDMA는 매체 접근 제어 프로토콜

---

### TDMA (시간 분할 다중접속)

**Time Division Multiple Access**

- 각 지국들은 **할당된 시간슬롯**을 사용하여 데이터를 전송
- **모든 지국 사이의 동기화**가 중요함

```
시간 슬롯:
┌────┬────┬────┬────┬────┬────┐
│ 지1│ 지2│ 지3│ 지4│ 지1│ 지2│  ← 순서대로 시간 슬롯 사용
└────┴────┴────┴────┴────┴────┘
          ↓ Common channel
```

> 6장 TDM과 개념 동일. 차이는 TDM은 물리층 다중화 기술, TDMA는 매체 접근 제어 프로토콜

### CDMA (코드 분할 다중접속)

**Code Division Multiple Access**

- 링크의 **전체 대역폭을 하나의 채널에서 점유**
- 모든 지국들은 **시분할 없이 동시에** 데이터를 송신 가능
- 각 지국에 고유한 **코드(칩 시퀀스)** 를 부여하여 구분

#### Figure 12.26 — Sharing channel in CDMA
```
지국 1 (코드 c1): d1 · c1 ──────────────────────────────→ ┐
지국 2 (코드 c2): d2 · c2 ──────────────────────────────→ ┤
지국 3 (코드 c3): d3 · c3 ──────────────────────────────→ ┤→ [d1·c1 + d2·c2 + d3·c3 + d4·c4] → Common channel
지국 4 (코드 c4): d4 · c4 ──────────────────────────────→ ┘

수신측: 수신 신호 × 해당 코드 → 해당 지국 데이터만 추출
```

> DSSS 기반. 6장에서 배운 확산 대역 방식을 매체 접근 제어에 적용한 것

#### CDMA 예시 (4개 지국, 코드 길이 4)
```
지국 1 코드: [+1 +1 +1 +1]
지국 2 코드: [+1 -1 +1 -1]
지국 3 코드: [+1 +1 -1 -1]
지국 4 코드: [+1 -1 -1 +1]

각 지국이 데이터 1 또는 0(=-1)을 전송할 때:
  전송 신호 = 데이터 × 코드

합산된 신호가 채널로 전송됨
수신측은 자신의 코드로 내적(inner product) 계산 → 자신의 데이터만 복원
```
